# AGENTS.md

VisualDSA is a pnpm monorepo with a Next.js web app and NestJS API, sharing types, utils, config, and a Prisma database package. All dependency versions are centralized via the `catalog:` protocol in `pnpm-workspace.yaml`.

## Architecture

```
apps/api       → NestJS API (@visualdsa/api), port 4000, Swagger at /api/docs
apps/web       → Next.js app (@visualdsa/web), port 3000, Turbopack, Tailwind v4
packages/db    → Prisma 7 with PostgreSQL (@visualdsa/db) — multi-file schema in prisma/schema/
packages/api-spec → Holds openapi.yaml; runs orval to generate the api-client
packages/api-client → Auto-generated by orval (react-query + custom fetch). DO NOT hand-edit generated/ dir
packages/types → Shared TypeScript interfaces (incl. the Step schema for algorithm steps)
packages/utils → Shared utility functions
packages/config → Shared app constants (ports, env flags)
packages/ui → Reusable React UI components (Tailwind) (@visualdsa/ui)
packages/visualizer → SVG visualization engine + playback controls (@visualdsa/visualizer)
packages/algorithms → Algorithm step generators yielding Step objects (@visualdsa/algorithms)
packages/content → MDX lesson content + lesson registry (@visualdsa/content)
packages/python-examples → Static Python code examples per topic (@visualdsa/python-examples)
configs/eslint   → @visualdsa/eslint — shared ESLint flat configs (base, nest, next)
configs/typescript → @visualdsa/typescript — shared tsconfigs (base, nest, next)
```

### Data Flow: Code Generation Pipeline

The project uses a code generation pipeline from API to client:

1. `apps/api` defines endpoints with `@nestjs/swagger` decorators (controllers, DTOs)
2. `pnpm run spec:generate` (in `@visualdsa/api`) boots the NestJS app with `SKIP_DB=true`, extracts OpenAPI spec, writes YAML to `packages/api-spec/openapi.yaml`
3. `pnpm --filter @visualdsa/api-spec run codegen` runs orval, which reads the YAML and generates React Query hooks into `packages/api-client/src/generated/`
4. `@visualdsa/web` imports `@visualdsa/api-client` to call the API

The one-liner for the full pipeline: `pnpm run codegen` (from root).

### API → Database Connection

- `@visualdsa/db` builds the Prisma client via `tsup` — the generated client lives in `packages/db/prisma/generated/`
- The API imports `PrismaClient` from `@visualdsa/db` and wraps it in `PrismaService` using the `@prisma/adapter-pg` (pg adapter, not the default Prisma query engine)
- `PrismaModule` is `@Global()` — inject `PrismaService` anywhere without importing the module

## Commands

All run from repo root unless noted. Turbo orchestrates across all workspaces.

```bash
pnpm dev            # Start all apps (api on :4000, web on :3000)
pnpm build          # Build all packages and apps
pnpm start          # Start production builds
pnpm lint           # ESLint check all
pnpm lint:fix       # ESLint fix all
pnpm format         # Prettier check all
pnpm format:fix     # Prettier write all
pnpm typecheck      # TypeScript check all
pnpm test           # Run tests (turbo test — depends on build first)
pnpm codegen        # Full pipeline: build db → generate spec → orval codegen
pnpm clean          # Remove all dist/node_modules
pnpm commit         # Commitizen interactive commit (cz-git)
```

## Setup (Fresh Checkout)

Generated code is gitignored, so a clean checkout needs regeneration before
`typecheck`/`build` pass:

1. `pnpm install`
2. `DATABASE_URL="postgresql://..." pnpm --filter @visualdsa/db db:generate` —
   generates the Prisma client into `packages/db/prisma/generated/`
3. `pnpm codegen` — builds `@visualdsa/db`, extracts the OpenAPI spec via the
   NestJS app (`SKIP_DB=true`), and runs orval to generate
   `packages/api-client/src/generated/`

After that, `pnpm build` then `pnpm typecheck` / `pnpm lint` / `pnpm format` pass.
**`pnpm typecheck` requires shared packages to be built first** — consumers import
each other via `package.json` `types` → `dist/` (e.g. `@visualdsa/algorithms`
imports `@visualdsa/types`), so run `pnpm build` before `pnpm typecheck` on a clean
checkout. The `@visualdsa/ui`, `visualizer`, `algorithms`, `content`, and
`python-examples` packages are currently stubs awaiting implementation per
`docs/deep-research-report.md`.

## Local Stack (Docker + PM2)

Local infra mirrors the verivox pattern: Postgres + pgAdmin in Docker, API + web
on the host (dev via `concurrently`, prod via PM2). See `docs/STARTUP.md`.

- **`compose.yaml`** — Postgres 16-alpine (host `5434`→`5432`, user/db
  `visualdsa`) + pgAdmin (`8123`, `/pgadmin`). Healthcheck + named volume.
- **`Makefile`** — `make setup` = `install → env → db-up → db-wait →
db-generate → db-push`. Plus `dev-stack`, `db-up/down/reset`, `docker-up`.
- **`ecosystem.config.cjs`** — PM2 fork-mode: `visualdsa-api`
  (`node apps/api/dist/main/main.js`, port 4000) + `visualdsa-web`
  (real Next.js CLI `next start`, port 3000).
- **`scripts/repo-pm2.mjs`** — pins `PM2_HOME` to `<repo>/.pm2` so
  `pnpm prod` / `pnpm prod:stop` share one daemon.
- **`scripts/kill-prod-ports.mjs`** — frees 4000 + 3000 (stops PM2, then
  SIGKILLs listeners). Used by `dev:stack`.

Key root scripts:

```bash
make setup          # one-shot DB + repo prep
pnpm dev:stack      # Postgres + schema + API + web (concurrently)
pnpm prod           # pnpm build + PM2 start (API + web)
pnpm prod:stop      # PM2 delete both
pnpm db:up          # docker compose up -d postgres
pnpm docker:up      # Postgres + pgAdmin
pnpm db:push        # prisma db push (schema -> DB)
```

Ports: web `3000`, API `4000` (Swagger at `/api/docs`), Postgres `5434`,
pgAdmin `8123`. `.env` files are copied from `*.example` by `make env`
(`packages/db`, `apps/api`, `apps/web`) — never overwritten.

## Conventions

### Package Naming

All packages use the `@visualdsa/` scope. References use `workspace:*` in package.json.

### Path Aliases

Both apps use `@/*` → `src/*`:

- `apps/api`: configured in `tsconfig.json` paths
- `apps/web`: configured in `tsconfig.json` paths

### Prisma Multi-File Schema

The Prisma schema is split across files in `packages/db/prisma/schema/`. The base `schema.prisma` defines the generator and datasource; model files (e.g., `user.prisma`) are merged by Prisma 7. The generated client outputs to `packages/db/prisma/generated/`.

### NestJS Patterns

- Entry point is `src/main/main.ts` (not the default `src/main.ts`)
- `createApp()` is exported separately — used by both `bootstrap()` and the spec generator
- Controllers register under `src/main/` grouped by domain (e.g., `src/main/auth/`)
- DTOs live alongside their controllers in `dto/` subdirectories
- Swagger decorators (`@ApiTags`, `@ApiOperation`, `@ApiResponse`, `@ApiProperty`) are required on all endpoints and DTOs — they drive the code generation pipeline
- The `src/common/` directory holds cross-cutting concerns (e.g., `prisma.service.ts`)

### Commit Convention

Commitlint enforces conventional commits. Allowed types: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, `revert`, `wip`, `release`. Use `pnpm commit` for interactive prompts.

### Pre-commit Hook

Husky runs `lint-staged` on pre-commit. Staged `*.{js,ts,jsx,tsx}` files get `eslint --fix` + `prettier --write`. Staged `*.{json,md}` get `prettier --write`.

### Style

- Prettier: single quotes, 2-space tabs, semicolons, 80 char print width
- ESLint: `no-console` is warn (console.warn/error/info allowed); unused vars with `_` prefix are ignored; `@typescript-eslint/no-explicit-any` is off
- Nest generation is configured with `"spec": false` — no test files are auto-generated

## Gotchas

- **Spec generation requires `SKIP_DB=true`**: The spec generator script (`apps/api/scripts/spec.ts`) sets `SKIP_DB=true` and a placeholder `DATABASE_URL` at the top before importing the app module. This prevents Prisma from trying to connect during spec generation. The `ConfigModule` in `AppModule` checks `process.env.SKIP_DB === 'true'` to skip `.env` loading.
- **Prisma generated output is not in the default location**: The generated client is at `packages/db/prisma/generated/` (not `node_modules/.prisma/client`). The `@visualdsa/db` package re-exports from `../prisma/generated/client`.
- **`@visualdsa/api-client` exports raw `.ts` source**: Its `package.json` exports map points to `./src/index.ts` directly (no build step for the client package — it's consumed as TypeScript by the Next.js app which compiles it).
- **`generated/` directories are gitignored**: Both `packages/db/prisma/generated/` and `packages/api-client/src/generated/` are generated code. Run `pnpm run codegen` after changing API endpoints.
- **Orval config has a hardcoded title transformer**: It rewrites the OpenAPI spec title to `'Api'` regardless of what NestJS produces. This is for orval's naming conventions.
- **`legacy-peer-deps=true`** is set in `.npmrc` — needed for some Prisma/NestJS peer dep conflicts.
- **No CI workflows yet**: The `.github/workflows/` directory exists but is empty. The `ci:check` and `ci:fix` scripts exist in root `package.json` but are not wired to any CI runner.
- **No test framework configured**: `pnpm test` runs `turbo run test` but no workspace has a test runner (jest, vitest, etc.) installed yet.

- `PORT` — API port (default: 4000)

**packages/db** (`.env`):

- `DATABASE_URL` — PostgreSQL connection string for Prisma CLI commands
